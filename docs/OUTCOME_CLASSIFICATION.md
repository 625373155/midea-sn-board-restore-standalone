# 结果分类

结果描述“现有证据支持什么”，不授权再次写入。只要写入保留已经建立，本事件就不能因超时、无 ACK、零字节或异常而重试。

| 状态 | 最低证据 | 后续动作 |
|---|---|---|
| `READ_ONLY_ALREADY_CORRECT` | 写入保留前，完整回复通过传输、解密、CRC/校验和和严格 SN 解码，结果精确等于目标 22 位 SN。 | 建立包内/全局永久 no-write 标记并停止。 |
| `READ_ONLY_INVALID_ENCODING` | 写入保留前，完整回复通过完整性检查，但 22 字节身份载荷不能严格解码为 22 位 ASCII 数字。 | 建立包内/全局 `READ-INVALID-DO-NOT-WRITE`，停止并交由授权维修复核。 |
| `READ_ONLY_MISMATCH` | 严格解码得到有效 22 位 SN，但不等于目标。 | 建立包内/全局 `READ-MISMATCH-DO-NOT-WRITE`，禁止覆盖并复核。 |
| `DIAGNOSTIC_ZERO_RESPONSE` | 只读诊断在有限窗口收到 0 字节，且没有传输错误和写入保留。 | 只记录零字节；它不证明板为空。只有全部资格门仍成立时才能考虑首次写入。 |
| `WRITE_RESERVED` | 包内和全局原子保留已创建，尚未确定发送是否开始。 | 将事件视为已消耗；恢复审计事实，但不删除锁。 |
| `WRITE_NOT_SENT_BUT_LOCKED` | 保留已创建，且本地审计确认本进程在 `NetworkStream.Write` 开始前失败。 | 保留锁；不得自动解锁或重试，只能人工代码/审计复核。 |
| `WRITE_SENT_READBACK_VERIFIED` | 恰好一次写路径已开始，随后严格读回精确目标 SN。 | 保存证据并停止；永不重写本事件。 |
| `WRITE_SENT_APP_VERIFIED` | 写路径可能已开始；冷启动后，已明确验证兼容展示关系的官方 App 显示目标身份且控制正常，无矛盾读回。 | 保存人工证据并停止；永不重写本事件。 |
| `WRITE_SENT_MISMATCH` | 写路径已开始后，严格读回有效非目标 SN，或可信 App 证据显示矛盾身份。 | 立即停止，保存原始证据并交由授权维修复核。 |
| `WRITE_SENT_UNKNOWN` | 写路径已开始或可能已开始，但没有精确目标读回，也没有合格 App 验证。包括部分发送、超时、断连、异常、无 ACK 或零字节。 | 不得重试；只允许冷启动、只读、官方 App 或授权售后诊断。 |

证据强度从高到低为：严格协议读回、满足全部条件的冷启动官方 App、完整 ACK/有效非目标读回、原始字节数与连接状态。TCP 连接和 ACK 都不证明写入成功；零字节既不证明空板，也不证明写入失败。
